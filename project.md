# 🃏 动物股东 — 项目描述文档

> 一款 3-7 人的在线卡牌桌游，玩家通过收集动物公司卡牌争夺大股东地位，利用反垄断机制和市场博弈获取最高收益。

---

## 一、游戏概述

### 1.1 核心玩法

玩家扮演投资者，通过摸牌或从市场获取卡牌，将卡牌放入自己的放置区以争夺各家"动物公司"的大股东地位。大股东拥有反垄断标记，在摸牌时可节省资金，但也限制了从市场获取同类卡牌的能力。游戏结束时，大股东向持有其公司卡牌的其他玩家收取罚金（且收益放大 3 倍），最终资金最多者获胜。

### 1.2 游戏人数

3 - 7 人

### 1.3 游戏时长

预计 10-20 分钟/局

## 三、完整游戏规则

### 3.1 游戏初始化

1. 将 45 张卡牌洗牌后，**随机移除 5 张**（所有玩家不可见），剩余 40 张组成牌堆。
2. 每位玩家从牌堆**发 3 张手牌**。
3. 每位玩家获得 **10 资金**。
4. 市场区域初始为空。
5. 随机决定起始玩家，按顺序轮流行动。

### 3.2 回合流程

每个玩家的回合分为两个阶段：

#### 阶段一：获取卡牌（二选一）

**选项 A：从牌堆摸牌**

- 玩家需要为市场上的**每张卡牌**支付 1 资金（资金放在对应的市场卡牌上）。
- **例外**：如果玩家拥有某家公司的反垄断标记，则无需为该公司的市场卡牌支付资金。
- 如果玩家资金不足以支付，则**不能选择摸牌**。
- 从牌堆顶部摸取 1 张牌加入手牌。

**选项 B：从市场获取卡牌**

- 玩家选取市场中的一张卡牌，将其加入手牌。
- 同时获取该卡牌上累积的所有资金。
- **限制**：不能选取自己拥有反垄断标记的公司的卡牌（规则 4）。

#### 阶段二：打出卡牌（二选一，必须执行）

从手牌中选择一张卡牌：

**选项 A：打出到市场**

- 将卡牌放入市场公共区域，初始资金为 0。
- **限制**：本轮从市场获取卡牌的玩家，不能将**相同公司**的卡牌打回市场（规则 3）。

**选项 B：放入自己的放置区**

- 将卡牌放入自己的放置区，公开可见。
- 放置后可能改变大股东归属（见 3.3）。

回合结束后，手牌数量恢复为 3 张。轮到下一位玩家。

### 3.3 大股东与反垄断标记

- 对于每家公司，**放置区中持有该公司卡牌最多的玩家**为大股东。
- 大股东自动拥有该公司的**反垄断标记**。
- **平局不更替**：当其他玩家的放置区中该公司卡牌数量追平大股东时，大股东身份**不发生改变**（先到先得）。
- 如果原大股东被超越，大股东易主。

**反垄断标记的效果：**

- ✅ 摸牌时，无需为市场中该公司的卡牌支付资金（省钱）。
- ❌ 不能从市场中选取该公司的卡牌（限制选择）。

### 3.4 游戏结束条件

当牌堆中**最后一张卡牌被摸取**，该玩家完成其完整回合后，游戏立即进入计分环节。

### 3.5 计分规则

对每家公司分别结算：

1. **判定大股东**：统计所有玩家（放置区 + 手牌）该公司的总卡牌数。如果持有最多卡牌的玩家**多于一位**（并列第一），则该公司**没有大股东**，跳过该公司的结算。

2. **罚金支付**：如果存在唯一大股东，其他所有玩家如果**持有**（手牌）或**放置区有**该公司的卡牌，每张卡牌需向大股东支付 **1 资金**。

3. **收益放大**：大股东每收到 1 资金的罚金，实际获得 **3 资金**（即支付方扣 1，大股东加 3）。

4. 所有公司结算完毕后，**资金最多的玩家获胜**。

> **注意**：计分时，"持有"包含手牌中的卡牌，不仅限于放置区。

### 3.6 规则边界案例

- 玩家资金可以为负数（计分阶段的罚金可能导致负数）。
- 资金不够摸牌时，只能选择从市场取牌。
- 如果市场为空，摸牌费用为 0。
- 初始状态下没有任何大股东和反垄断标记。
- 移除的 5 张牌全局不可见，增加博弈不确定性。

## 四、技术架构

### 4.1 项目定位

Web 端在线卡牌游戏，支持通过房间号组局，WebSocket 实时通信。当前阶段为本地开发调试（多窗口模拟多玩家）。

### 4.2 技术选型

| 层级 | 技术 |
|------|------|
| 前端 | HTML + Tailwind CSS + Vanilla JS（或轻量框架） |
| 后端 | Python + FastAPI / uvicorn |
| 通信 | WebSocket（实时双向通信） |
| 部署 | 本地开发优先，后续可部署至云端 |

### 4.3 项目结构

```
/
├── server/
│   ├── main.py               # 服务入口，FastAPI + WebSocket
│   ├── game/
│   │   ├── manager.py        # 房间管理、生命周期
│   │   ├── state.py          # 核心游戏状态与规则引擎
│   │   └── deck.py           # 牌堆管理（洗牌、发牌、移除）
│   └── ws/
│       └── handler.py        # WebSocket 消息路由与处理
├── public/
│   ├── index.html            # 首页（创建/加入房间）
│   ├── room.html             # 房间等待页（等待玩家、准备开始）
│   ├── game.html             # 游戏主页面
│   ├── css/
│   │   └── style.css         # 自定义样式（配合 Tailwind）
│   ├── js/
│   │   ├── websocket.js      # WebSocket 连接管理
│   │   ├── room.js           # 房间页逻辑
│   │   ├── game.js           # 游戏主逻辑 & 渲染
│   │   └── ui.js             # UI 交互组件
│   └── assets/
│       └── ...               # 图标、图片资源
└── requirements.txt          # Python 依赖（fastapi, uvicorn 等）
```

### 4.4 通信协议（WebSocket 消息设计）

#### 客户端 → 服务端

| 消息类型 | 说明 | 关键字段 |
|----------|------|----------|
| `create_room` | 创建房间 | `playerName` |
| `join_room` | 加入房间 | `roomCode`, `playerName` |
| `player_ready` | 玩家准备 | — |
| `start_game` | 房主开始游戏 | — |
| `draw_card` | 从牌堆摸牌 | — |
| `pick_market` | 从市场取牌 | `cardIndex` |
| `play_to_market` | 打出到市场 | `handIndex` |
| `play_to_area` | 放入放置区 | `handIndex` |

#### 服务端 → 客户端

| 消息类型 | 说明 | 关键字段 |
|----------|------|----------|
| `room_created` | 房间创建成功 | `roomCode` |
| `room_update` | 房间玩家变动 | `players[]`, `hostId` |
| `game_start` | 游戏开始 | 初始状态 |
| `game_state` | 游戏状态更新 | 完整/差量状态 |
| `your_turn` | 轮到你 | `phase`（draw/play） |
| `turn_update` | 其他玩家行动通知 | `action`, `playerId` |
| `game_end` | 游戏结束 | `scores[]`, `winner` |
| `error` | 错误提示 | `message` |
| `player_disconnected` | 玩家断线 | `playerId` |
| `game_aborted` | 游戏因断线终止 | `reason` |

### 4.5 状态管理

#### 服务端维护的完整游戏状态

```python
{
    "room_code": "A1B2C3",
    "phase": "playing",            # waiting / playing / scoring / ended
    "current_player_index": 0,
    "turn_phase": "draw",          # draw（获取卡牌阶段） / play（打出卡牌阶段）
    "deck": [...],                 # 剩余牌堆（对玩家隐藏）
    "removed_cards": [...],        # 开局移除的 5 张（全局隐藏）
    "market": [                    # 市场卡牌
        {"card": "🐶", "coins": 3}
    ],
    "players": [
        {
            "id": "ws-id",
            "name": "玩家1",
            "hand": ["🦒", "🐶", "🐘"],       # 手牌（仅自己可见）
            "area": {"🦒": 2, "🐶": 1, ...},   # 放置区（公开）
            "coins": 10,
            "anti_monopoly": ["🦒"],            # 持有的反垄断标记
        }
    ],
    "major_shareholders": {        # 各公司当前大股东
        "🦒": "player-id-1",
        "🐶": None,               # 暂无
        ...
    },
    "turn_context": {              # 当前回合临时上下文
        "picked_from_market": "🐶",  # 本轮从市场取的卡牌类型（用于规则3）
    },
    "last_card_drawn": False,      # 是否已摸到最后一张牌
}
```

#### 客户端接收的状态（信息隐藏）

- 自己的手牌：可见
- 其他玩家的手牌：仅显示数量
- 牌堆：仅显示剩余张数
- 移除的牌：不可见
- 其他所有信息（市场、放置区、资金、反垄断标记）：全部公开

### 4.6 前后端交互流程

#### 核心广播原则

服务端是唯一权威状态源。每次状态变更后，服务端向**所有连接的客户端**广播 `game_state`，但每个客户端收到的是**各自的私有视角**：

| 信息类型 | 当前玩家 | 其他玩家 |
|----------|---------|---------|
| 自己的手牌 | ✅ 具体内容 | ❌ 仅看到数量 |
| 其他人手牌 | ❌ 仅看到数量 | ❌ 仅看到数量 |
| 牌堆内容 | ❌ 仅看到剩余数 | ❌ 仅看到剩余数 |
| 市场卡牌 | ✅ 完整 | ✅ 完整 |
| 放置区 | ✅ 完整 | ✅ 完整 |
| 资金 | ✅ 完整 | ✅ 完整 |
| 反垄断标记 | ✅ 完整 | ✅ 完整 |
| 移除的5张牌 | ❌ | ❌ |

#### 流程一：房间生命周期

```
玩家A → 服务端: create_room(name)
服务端 → 玩家A: room_created(code)

玩家B → 服务端: join_room(code, name)
服务端 → 全员广播: room_update(players=[A,B])

玩家A → 服务端: start_game  (需≥3人)
服务端 → 逐个发送: game_start(各自手牌, 公共状态)
```

#### 流程二：摸牌回合

```
服务端 → 当前玩家: your_turn(phase="draw")
当前玩家 → 服务端: draw_card

服务端校验:
  - 资金 >= 市场卡数 - 反垄断豁免数
  - 扣资金，给市场每张卡加1资金
  - 牌堆顶摸1张

服务端 → 当前玩家: game_state(手牌+1, 资金变化, 新牌内容)
服务端 → 其他玩家: game_state(手牌数+1, 资金变化, 市场卡资金更新, 牌堆数-1)

服务端 → 当前玩家: your_turn(phase="play")
当前玩家 → 服务端: play_to_market(handIndex) 或 play_to_area(handIndex)

服务端校验:
  - play_to_market: 检查本轮是否从市场取过同类卡
  - play_to_area: 重新计算大股东和反垄断标记

服务端 → 全员广播: game_state(更新后状态)
服务端 → 下一位玩家: your_turn(phase="draw")
```

#### 流程三：市场取牌回合

```
服务端 → 当前玩家: your_turn(phase="draw")
当前玩家 → 服务端: pick_market(cardIndex)

服务端校验:
  - 该玩家不持有该卡公司的反垄断标记
  - 卡牌加入手牌，卡上资金归玩家
  - 记录本轮取牌类型(turnContext)

服务端 → 当前玩家: game_state(手牌+1, 资金+N, 市场移除该卡)
服务端 → 其他玩家: game_state(手牌数+1, 资金+N, 市场移除该卡)

(后续打出阶段同流程二)
```

#### 流程四：游戏结束与计分

```
当前玩家摸到牌堆最后一张 → 完成打出阶段后触发

服务端逐公司结算:
  1. 统计每位玩家该公司总卡牌数(手牌+放置区)
  2. 判定大股东(唯一最多者)
  3. 非大股东每张卡支付1资金，大股东每收1得3

服务端 → 全员广播: game_end(
  scores: 每位玩家最终资金,
  details: 每家公司的结算明细(大股东、罚金流向),
  winner: 获胜者,
  revealed_hands: 所有人手牌公开
)
```

#### 流程五：断线处理

```
等待阶段断线:
  服务端 → 其余玩家: room_update(移除该玩家)

游戏中断线:
  服务端 → 其余玩家: player_disconnected + game_aborted
  (当前版本不支持重连，游戏终止)
```

#### 前端状态合并策略

客户端收到 `game_state` 后的处理流程：

1. 更新全局公共状态（市场、所有放置区、所有资金、反垄断标记、牌堆剩余数）
2. 更新自己的手牌（仅在 state 中包含时）
3. 更新其他玩家手牌数量
4. 根据 `current_player_index` 判断是否轮到自己，控制 UI 交互状态
5. 根据 `turn_phase` 显示/隐藏操作栏
6. 检查操作合法性，置灰不可用按钮

## 五、界面设计

### 5.1 页面结构

共 3 个页面：

**① index.html — 首页**
- 游戏标题 & Logo
- 「创建房间」按钮 → 输入昵称 → 创建后跳转 room.html
- 「加入房间」→ 输入房间码 + 昵称 → 跳转 room.html
- 支持通过 URL 参数直接加入（如 `room.html?code=A1B2C3`）

**② room.html — 房间等待页**
- 显示房间码（可复制/分享链接）
- 玩家列表（头像 + 昵称，房主标识）
- 房主可见「开始游戏」按钮（至少 3 人时激活）
- 玩家加入/离开实时更新

**③ game.html — 游戏主页面**
- 核心游戏界面（详见 5.2）

### 5.2 游戏界面布局

设计风格：类似桌游的俯视视角，温暖、卡通化配色。

#### 整体布局

```
┌──────────────────────────────────────────────────┐
│              上方玩家区（对面玩家）                  │
│  [玩家2]    [玩家3]    [玩家4]                     │
├──────────────────────────────────────────────────┤
│                                                    │
│ 左  ┌──────────────────────────────────┐   右      │
│ 侧  │                                  │   侧      │
│ 玩  │         中央市场区域              │   玩      │
│ 家  │    (牌堆 + 市场卡牌栅格)          │   家      │
│     │                                  │          │
│ [5] └──────────────────────────────────┘   [6]    │
│ [7]                                               │
├──────────────────────────────────────────────────┤
│                  操作选项栏                         │
│            [打出到市场] [放入放置区]                 │
├──────────────────────────────────────────────────┤
│              自己的手牌区（最多4张）                 │
│          [牌1]  [牌2]  [牌3]  [牌4]               │
├──────────────────────────────────────────────────┤
│           自己的信息区（头像+放置区）                │

└──────────────────────────────────────────────────┘
```

#### 玩家信息展示（每个玩家面板）

- 头像 + 昵称
- 资金数字（💰 图标 + 数字）
- 放置区：2×3 或 3×2 栅格
  - 按卡牌稀有度顺序固定排列：🦒(5) → 🐶(6) → 🦢(7) → 🐙(8) → 🦛(9) → 🐘(10)
  - 每格显示：公司图标 + 已放置数量
  - 数量为 0 时灰显
  - 持有反垄断标记的格子：加背景色高亮（如金色/橙色边框或底色）

#### 中央市场区域

- 默认 3 列 × 2 行的栅格布局
- **第一格固定为牌堆**：
  - 显示卡牌背面样式
  - 中间显示剩余张数（如 "余 23 张"）
  - 底部小条显示摸牌费用（如 "-3" 表示需支付 3 资金）
- **其余格为市场卡牌**：
  - 显示卡牌正面（公司图标）
  - 底部小条显示卡牌上累积的资金（如 "+2" 表示可获取 2 资金）
- **溢出处理**（市场卡牌 ≥ 5 张时）：
  - 最后一格显示"更多"（三个点图标）
  - 点击后弹出浮动层，单行横向排列所有市场卡牌
  - 第一个仍为牌堆，其余按出牌顺序排列
  - 支持左右滑动
  - 点击空白区域关闭浮动层
  - 点击卡牌可执行摸牌/取牌操作

#### 操作选项栏

- 默认隐藏
- 进入"打出卡牌"阶段时显示两个按钮：「打出到市场」「放入放置区」
- 如果选择的操作不合法（如：本轮从市场拿了🐶，又想把🐶打回市场），对应按钮置灰 + 不可点击

#### 卡牌交互

- 摸牌/取牌阶段：
  - 第一次点击卡牌：显示选中光标/高亮
  - 再次点击同一张：确认执行操作
  - 点击其他卡牌：切换选中
- 打出卡牌阶段：
  - 点击手牌选中 → 点击「打出」或「放置」按钮执行

### 5.3 设计风格

- 整体色调：温暖、柔和，类似实体桌游的桌面质感
- 卡牌风格：圆角卡片，轻微阴影，动物 Emoji 为主视觉
- 字体：清晰易读，中文优先
- 动画：卡牌移动、翻转时有简单过渡动画
- 适配：优先 iPad/桌面端横屏，移动端竖屏可适当调整布局

## 六、用户流程

### 6.1 创建并开始游戏

```
首页 → 点击"创建房间" → 输入昵称 → 进入房间等待页
→ 分享房间码给朋友 → 等待玩家加入（3-7人）
→ 房主点击"开始游戏" → 进入游戏页面
```

### 6.2 加入游戏

```
收到房间码/链接 → 首页输入房间码+昵称（或直接点链接）
→ 进入房间等待页 → 等待房主开始 → 进入游戏页面
```

### 6.3 单回合流程

```
轮到你 → 阶段一（获取卡牌）:
  ├─ 点击牌堆（摸牌）→ 确认 → 支付资金 → 获得1张牌
  └─ 点击市场卡牌（取牌）→ 确认 → 获得卡牌+上面的资金

→ 阶段二（打出卡牌）:
  → 选择一张手牌 → 点击"打出到市场"或"放入放置区"
  → 回合结束，轮到下一位玩家
```

### 6.4 游戏结束

```
最后一张牌被摸取 → 当前玩家完成回合
→ 计分动画：逐公司结算，展示大股东和罚金流向
→ 显示最终排名和资金数 → 可选"再来一局"或"返回首页"
```

## 八、已知设计决策记录

| 编号 | 决策 | 原因 |
|------|------|------|
| D1 | 本轮不可打回从市场获取的同类卡牌 | 防止无意义循环操作 |
| D2 | 每轮必须打出一张牌 | 保持手牌恒定为3张，简化博弈 |
| D3 | 市场卡牌资金随取走全部带走 | 给市场卡牌增加价值，鼓励等待 |
| D4 | 手牌计入计分持有 | 增加博弈深度，手牌也是风险 |
| D5 | 暂不支持跳过回合 | 理论上始终有可执行的操作 |
| D6 | 大股东平局不更替 | 保护先发优势，增加策略性 |
| D7 | 大股东收益3倍放大 | 使大股东地位有足够吸引力 |
| D8 | 开局随机移除5张 | 制造信息不完美，增加不确定性 |
